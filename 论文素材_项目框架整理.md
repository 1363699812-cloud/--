## 三、用户角色与权限设计（RBAC模型）

### 3.1 四种角色定义

| 角色标识 | 角色名称 | 权限范围 |
|---------|---------|---------|
| admin | 系统管理员 | 全部功能（含用户管理、操作日志） |
| warehouse_keeper | 仓库管理员 | 基础数据管理 + 全部库存业务 + 报表统计 |
| purchaser | 采购员 | 供应商管理 + 入库单管理 + 库存预警查看 |
| seller | 销售员 | 客户管理 + 出库单管理 |

### 3.2 权限控制实现方式

1. **后端接口级**：自定义 `@RequireRole` 注解 + JWT拦截器校验角色
2. **前端路由级**：Vue Router `meta.roles` 字段 + `beforeEach` 路由守卫
3. **前端菜单级**：根据角色动态显示/隐藏侧边栏菜单项

## 五、核心业务流程设计

### 5.1 单据状态机（通用流程）

```
                        驳回(-1)
                     ┌──────────┐
                     │          ▼
创建(0:待审核) ───→ 审核(1:已审核) ───→ 完成(2:已完成)
```

适用于：入库单、出库单、调拨单、报损单。盘点单略有不同（0进行中→1已完成）。

### 5.2 入库流程

```
[采购员]创建入库单(选仓库+供应商+物资明细)
    → 状态=0(待审核)
    → [管理员/仓管]审核入库单
    → 状态=1(已审核)
    → [仓管]确认到货，执行入库
    → 后端逻辑：
        1. 遍历入库明细
        2. inventory表: quantity += item.quantity（若无记录则INSERT）
        3. inventory_transaction表: 插入IN类型流水记录
        4. inbound_order状态 → 2(已完成)
```

### 5.3 出库流程（含库存锁定机制）

```
[销售员]创建出库单(选仓库+客户+物资明细)
    → 状态=0(待审核)
    → [管理员/仓管]审核出库单
    → 后端逻辑：
        1. 检查所有物资库存是否充足（quantity - lock_quantity >= 需求量）
        2. 锁定库存: lock_quantity += item.quantity
        3. outbound_order状态 → 1(已审核)
    → [仓管]确认发货，执行出库
    → 后端逻辑：
        1. inventory表: quantity -= item.quantity, lock_quantity -= item.quantity
        2. inventory_transaction表: 插入OUT类型流水记录
        3. outbound_order状态 → 2(已完成)
    
    若驳回：
        → 解锁库存: lock_quantity -= item.quantity
        → outbound_order状态 → -1(已驳回)
```

### 5.4 调拨流程

```
[仓管]创建调拨单(源仓库→目标仓库+物资明细)
    → 状态=0
    → 审核: 检查并锁定源仓库库存
    → 完成调拨:
        1. 源仓库: quantity -= item.quantity, 生成TRANSFER_OUT流水
        2. 目标仓库: quantity += item.quantity, 生成TRANSFER_IN流水
        （若目标仓库无该物资库存记录则自动创建）
```

### 5.5 报损/报废流程

```
[仓管]创建报损单(选仓库+类型DAMAGE/SCRAP+物资明细+原因)
    → 审核 → 完成: inventory.quantity -= item.quantity, 生成DAMAGE/SCRAP流水
```

### 5.6 库存盘点流程

```
[仓管]创建盘点单(选择仓库)
    → 系统自动加载仓库所有物资，填充system_quantity = 当前库存
    → 人工逐一录入actual_quantity（实盘数量）
    → 完成盘点:
        1. 计算 diff_quantity = actual_quantity - system_quantity
        2. 若diff > 0: inventory += diff（盘盈）
        3. 若diff < 0: inventory += diff（盘亏，即减少）
        4. 生成STOCKTAKE类型库存流水
```

### 5.7 库存预警机制

```
预警条件: inventory.quantity < material.safety_stock
实现方式:
    1. 后端SQL: JOIN inventory和material表，筛选quantity < safety_stock的记录
    2. 前端展示: 预警物资列表 + 缺口数量(safety_stock - quantity)
    3. 仪表板: 预警总数卡片 + 快捷入口
预警等级:
    - 红色(严重): 库存 ≤ 0
    - 橙色(预警): 0 < 库存 < 安全库存
```

## 六、关键技术实现细节

### 6.1 身份认证与鉴权（JWT）

- **Token生成**：用户登录成功后，后端生成JWT Token（有效期24小时），包含userId、username、role
- **Token传递**：前端将Token存入Cookie，每次请求通过Authorization请求头传递（Bearer Token格式）
- **Token验证**：`JwtInterceptor`拦截器在每次请求前验证Token有效性，解析用户信息存入request属性
- **放行路径**：`/api/user/login` 和 `/api/user/register` 不需要Token

### 6.2 库存并发控制（乐观锁）

- **inventory表**设有`version`字段
- 使用MyBatis-Plus的`@Version`注解 + `OptimisticLockerInnerInterceptor`插件

```vue
    } finally {
      loading.value = false
    }
```

```java
    QueryWrapper<User> wrapper = new QueryWrapper<>();
    wrapper.eq("username", loginDTO.getUsername());
    User user = userService.getOne(wrapper);

    if (user == null) return Result.error("用户不存在");
    if (!user.getPassword().equals(loginDTO.getPassword())) return Result.error("密码错误");
    if (user.getStatus() != null && user.getStatus() == 0) return Result.error("账号已被禁用");

    // 生成Token
    String token = JwtUtils.generateToken(user.getId(), user.getUsername(), user.getRole());

    LoginVO loginVO = new LoginVO();
    loginVO.setToken(token);
    loginVO.setId(user.getId());
    loginVO.setUsername(user.getUsername());
    loginVO.setRealName(user.getRealName());
    loginVO.setRole(user.getRole());

    return Result.success(loginVO);
}
```

```java
    return Jwts.builder()
            .setSubject(String.valueOf(userId))
            .claim("username", username)
            .claim("role", role)
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + EXPIRATION))
            .signWith(SECRET_KEY, SignatureAlgorithm.HS256)
            .compact();
```

## 九、统计报表模块详细说明（论文重点）

### 9.1 仪表板（Dashboard）

**数据组成**：
- 统计卡片：物资总数(material COUNT)、仓库总数(warehouse COUNT)、库存总量(inventory SUM(quantity))、预警数量(quantity < safety_stock COUNT)
- 趋势图（7天）：每日入库数量、每日出库数量（ECharts折线图）
- 快捷信息：待审核入库单数、待审核出库单数

### 9.2 ABC分析

**分析方法**：
1. 查询所有物资的库存总金额（quantity × unit_price）
2. 按金额降序排列
3. 计算累计金额占比
4. 分类规则：
   - A类：累计占比 ≤ 80%（高价值物资，重点管理）
   - B类：80% < 累计占比 ≤ 95%（中等价值）
   - C类：累计占比 > 95%（低价值物资）

### 9.3 库存周转率

**计算公式**：`周转率 = 期间出库总量 / 平均库存`
- 平均库存 = (期初库存 + 期末库存) / 2
- 支持按日期范围筛选

  ```java
  @Transactional
  public boolean increaseQuantity(Long warehouseId, Long materialId, Integer quantity) {
    if (quantity <= 0) return false;
    Inventory inventory = this.getByWarehouseAndMaterial(warehouseId, materialId);
    if (inventory == null) {
      inventory = new Inventory();
      inventory.setWarehouseId(warehouseId);
      inventory.setMaterialId(materialId);
      inventory.setQuantity(quantity);
      inventory.setLockQuantity(0);
      inventory.setVersion(0L);
      return this.save(inventory);
    } else {
      inventory.setQuantity(inventory.getQuantity() + quantity);
      return this.updateById(inventory);
    }
  }

  @Transactional
  public boolean decreaseQuantity(Long warehouseId, Long materialId, Integer quantity) {
    if (quantity <= 0) return false;
    Inventory inventory = this.getByWarehouseAndMaterial(warehouseId, materialId);
    if (inventory == null || inventory.getQuantity() < quantity) return false;
    inventory.setQuantity(inventory.getQuantity() - quantity);
    return this.updateById(inventory);
  }

  public boolean checkStockSufficient(Long warehouseId, Long materialId, Integer requiredQuantity) {
    if (requiredQuantity <= 0) return true;
    Inventory inventory = this.getByWarehouseAndMaterial(warehouseId, materialId);
    if (inventory == null) return false;
    int available = inventory.getQuantity() - (inventory.getLockQuantity() != null ? inventory.getLockQuantity() : 0);
    return available >= requiredQuantity;
  }

  @Transactional
  public boolean lockQuantity(Long warehouseId, Long materialId, Integer quantity) {
    if (quantity <= 0) return false;
    Inventory inventory = this.getByWarehouseAndMaterial(warehouseId, materialId);
    if (inventory == null) return false;
    int available = inventory.getQuantity() - (inventory.getLockQuantity() != null ? inventory.getLockQuantity() : 0);
    if (available < quantity) return false;
    inventory.setLockQuantity((inventory.getLockQuantity() != null ? inventory.getLockQuantity() : 0) + quantity);
    return this.updateById(inventory);
  }
  ```

  说明：`lockQuantity` 在出库审核时调用，`decreaseQuantity` 在完成出库时调用；两者均通过乐观锁保证并发安全。

## 十四、第一章完整版（绪论）

### 1.1 研究背景

#### 1.1.1 A公司仓储管理的现实困境

A公司是一家典型的中小型制造企业，其物资管理长期沿用纸质单据登记与Excel台账记录相结合的传统模式。在业务规模较小、物资品种有限、仓库结构简单的起步阶段，这种方式尚能勉强维持日常运转。然而随着企业的发展，物资品种从最初的数十种逐步增加到数百种，仓库也由单一库房扩展为主仓库与备用仓库并行，传统管理方式的弊端日益显现。

在入库环节，采购物资到货后，仓管人员需要逐项核对送货单、手工填写入库凭证并录入Excel表格。这一过程耗时长、易出错，经常出现漏记某种物资、数量填写错误或重复登记等情况。供应商实际到货时间与系统中登记时间之间存在明显的时间差，导致采购部门无法准确掌握订单的到货状态。

在出库环节，生产部门或销售部门的领用需求通常通过电话或口头沟通传递给仓管人员。由于缺乏实时的库存预检机制，经常出现"开出出库单后才发现库存不足"的情况，不仅耽误生产进度，还可能导致客户订单不能按时交付。出库记录的延迟更新进一步加剧了库存数据的滞后性。

在盘点环节，A公司每季度组织一次全面盘点，但由于日常出入库记录本身就不够准确，盘点结果与账面数据之间存在较大差异，账实不符率长期居高不下。面对差异数据，管理人员很难追溯具体的原因——究竟是入库少记、出库多记，还是物资丢失或损坏，往往无从查证。

#### 1.1.2 传统管理模式的深层局限

上述具体问题反映出传统仓储管理在三个层面存在系统性的局限。

首先是库存数据的一致性和实时性问题。由于入库、出库、调拨、报损等业务操作之间缺乏统一的数据同步机制，人工登记不可避免地存在漏记、错记、重复记账等情况。管理者在月底或季度末查看库存报表时，拿到的数据实际上已经滞后了数周甚至更久，基于滞后信息做出的采购决策和发货安排存在较大的偏差风险。当实际库存与账面库存存在显著差异时，企业可能面临缺货停工、重复采购、物资积压乃至客户交付延期等一系列连锁反应。

其次是业务流程的规范性不足。A公司的仓储活动长期依赖个人经验而非制度化的流程约束。入库时，并非每次都会核对供应商信息和采购订单；出库时，领用审批往往流于形式，没有强制性的库存充足性校验；物资发生损坏或过期需要处理时，也没有固定的报损记录和审批流程。一旦发生数量差异或责任争议，企业很难通过完整的历史记录来还原事件过程、界定责任归属。

最后是统计分析能力的缺失。A公司虽然积累了大量出入库原始数据，但这些数据以孤立的、静态的形式分散在各个Excel文件和工作表中，不能直接汇总为趋势图、分类报表或预警信息。管理者想要了解某个物资的库存是否合理、哪些物资占用资金最多、库存周转是否健康，只能依靠主观经验做粗略判断，缺乏量化的数据依据。这种"依靠经验而非依靠数据"的管理方式，使得库存结构优化不足、重点物资识别不清、资金占用控制效果不理想。

#### 1.1.3 仓储管理信息化的必然选择

从更宏观的视角来看，仓储并非企业运营中的孤立环节，而是采购管理、库存控制、生产供应保障与成本管理的交汇之处。物资的每一次入库、出库、调拨都同时影响着资金占用水平、供应链响应速度和客户服务水平。只有将仓储业务纳入信息系统进行统一管理，企业才能实现采购计划、库存状态和出库需求之间的有效衔接，从整体上提升供应链的协调效率。

随着Web开发技术、前后端分离架构和轻量级开源框架的日益成熟，中小企业构建专用仓储管理系统的技术门槛已经大幅降低。与动辄耗资数百万、实施周期以年计的大型ERP系统相比，基于Spring Boot和Vue.js等主流技术栈开发的轻量级WMS具有开发周期短、定制灵活性高、运维成本低的显著优势，非常适合中小企业在局部业务场景中快速落地。

从数字化转型的战略层面看，仓储管理作为企业基础数据最为集中、业务流程最为清晰的领域之一，是推进信息化改造的理想切入点。通过建设仓储管理系统，企业不仅能够解决当前的库存管理痛点，还能为后续向采购协同、财务集成、条码追溯乃至智能仓储方向扩展奠定数据基础和流程框架。

### 1.2 研究意义

#### 1.2.1 提升仓储业务效率与数据准确性

仓储管理信息化的首要价值在于用系统化的操作流程替代零散的人工处理，从而提升业务效率并降低差错率。通过将物资编码、仓库信息、供应商与客户档案维护在统一的基础数据模块中，业务人员在创建入库单或出库单时可以通过下拉选择、模糊搜索等方式快速定位数据，避免了手工抄写编码和名称的繁琐过程。同时，系统在单据提交时自动进行必要的数据校验——例如出库审核时强制检查库存是否充足、物资编码是否重复——从根本上减少了人工操作中的低级错误。

在库存更新方面，系统将入库完成、出库完成、调拨完成、报损完成等业务操作与库存数量的增减紧密绑定在同一数据库事务中，保证了"单据状态变更"与"库存数量变化"的严格同步。系统的乐观锁机制进一步降低了并发更新场景下的数据覆盖风险。这种自动化、事务级的数据一致性保障，是人工台账或Excel表格无法实现的。

#### 1.2.2 构建可量化可追溯的库存管控体系

信息系统的核心优势不仅在于替代纸质记录，更在于它能够将原本分散、隐性的业务信息转化为集中、显性的可分析数据。本系统以数据库为核心，将物资的基础属性（编码、名称、分类、规格、单位）、库存状态（当前库存、锁定库存、安全库存阈值、补货点、经济订货批量）、业务单据（入库单、出库单、调拨单、报损单、盘点单）以及库存变动流水统一存储和关联管理。

在这一数据基础之上，系统提供了多个维度的管理能力：实时库存查询使仓管人员随时了解每种物资在哪个仓库有多少可用数量；库存预警功能自动对低于安全库存阈值的物资发出警示，将风险识别从事后补救前移到事前预防；ABC分类分析帮助管理者快速识别高价值的关键物资，将有限的管理精力集中在最重要的品类上；库存周转率分析则从流动效率的角度评估库存资金的使用效果，为采购优化和库存结构调整提供量化依据。

此外，每一笔库存变动都以库存流水记录的形式持久化保存，包含变化的仓库、物资、数量、类型、来源单据以及操作人信息。这种全维度的操作留痕，使系统具备了较强的可追溯性和可审计性，为问题排查和责任界定提供了可靠的数据基础。

#### 1.2.3 为中小企业仓储信息化提供可参照的实践案例

从技术实现的角度看，本系统采用的Spring Boot + Vue.js前后端分离方案、MyBatis-Plus数据持久层、JWT无状态认证和AOP操作日志等技术组合，均为当前业界广泛使用且社区活跃的开源技术。系统遵循典型的控制层—业务层—数据访问层三层架构组织代码，模块划分清晰，职责边界明确，便于后续的功能迭代和维护扩展。

从业务设计的角度看，本系统将库存控制的数学模型（经济订货批量、安全库存、补货点）直接映射到数据库表结构和业务服务层中，使系统不仅是一个"记录出入库数据的电子台账"，更是"将管理理论嵌入运行逻辑的决策支持工具"。这种"模型驱动设计"的思路，为同类中小企业的仓储信息化建设提供了可参考的方法论和实践经验。

### 1.3 国内外研究现状

仓储管理系统（Warehouse Management System, WMS）的研究与实践经历了从手工记账到电子化记录、从单一功能到集成化平台、从作业执行到智能决策的演进过程。

在理论层面，库存控制一直是管理科学的重要分支。经济订货批量模型由Harris于1913年提出，后经Wilson推广，至今仍是确定最优订货批量的经典方法。Silver和Peterson在其著作中系统阐述了(s,Q)策略和周期性盘点策略，为随机需求环境下的库存补货规则提供了严格的理论框架。ABC分类法基于帕累托原理，将物资按价值贡献度分为A类（重点管控，约占80%价值）、B类（常规管理）和C类（简化管理），为企业合理分配管理资源提供了简洁有效的方法。安全库存模型通过服务水平和需求波动的量化分析，确定了应对不确定性所需的最低库存缓冲量。这些经典理论构成了现代仓储管理系统设计的理论基础。

在系统实践层面，国外较早形成了较为成熟的WMS产品体系。以SAP Extended Warehouse Management和Oracle WMS为代表的大型商业系统在流程集成、作业精细化、数据分析等方面具有显著优势，但其高昂的实施成本、漫长的部署周期和复杂的配置要求，使其主要适用于大型集团企业或物流中心。中小企业在预算有限、IT能力不足的情况下，往往难以承受重型商业软件的全套导入。

国内关于仓储管理系统的研究则更多聚焦于满足中小企业实际需求的轻量化方案。戴定一指出，WMS的核心价值在于通过信息系统强化仓库现场的作业控制和库存过程管理，提高物流运作效率并减少人为误差。曾宇容等基于条码技术探讨了现代仓储系统的设计思路，认为自动识别与数据采集是提升仓储作业效率的重要方向。张桂强、王改样、史鑫等学者从进销存的视角研究了企业管理系统的设计与实现，普遍认为业务单据的规范化和数据库的合理设计是系统成功的关键。

在技术架构方面，企业信息系统经历了从客户端/服务器（C/S）架构向浏览器/服务器（B/S）架构的转变。C/S架构需要在每台客户端安装专用程序，升级维护成本高；B/S架构以浏览器为统一入口，部署灵活、跨平台性强，更适合中小企业环境。近年来，前后端分离开发模式已成为Web应用的主流选择，前端负责页面渲染与用户交互，后端负责业务逻辑与数据持久化，通过RESTful API进行数据交换，系统的可维护性和扩展性得到显著提升。

从研究趋势来看，仓储管理系统的研究重心正在从单纯的"数据记录"转向"智能决策支持"。库存预警、ABC分类分析、库存周转率乃至基于机器学习的需求预测等功能被逐步纳入系统的设计范围。然而，现有的大多数研究仍然将数学模型与系统实现视为两个相对独立的层面——模型在论文的理论部分被阐述，但在实际系统的数据库设计和业务代码中，这些模型往往没有显式的对应关系。专门面向中小企业、强调"数学模型直接驱动数据库表结构设计"的仓储系统研究还相对薄弱。本文正是在这一背景下，将库存控制数学模型（目标函数、EOQ、安全库存、补货点）系统地映射到关系数据库的表结构和字段中，使数学上的优化目标与工程上的数据模型形成一一对应，从而在系统层面实现"模型驱动管理"的设计理念。

### 1.4 研究思路

本文围绕A公司物资仓储管理业务，遵循"理论准备—问题分析—模型建立—系统设计—开发实现—测试验证"的技术路线逐步展开研究。

首先，通过文献调研系统梳理仓储管理理论（库存控制模型、安全库存理论、ABC分类法）和系统开发技术（Spring Boot、Vue.js、MyBatis-Plus、JWT）的现状与发展趋势，为后续工作奠定理论和技术基础。

其次，深入分析A公司物资管理流程中的具体问题，识别出库存数据一致性差、业务流程不规范、决策分析能力弱三个核心痛点，明确系统需要解决的关键问题。在此基础上，建立以总库存成本最小化为目标的数学优化模型，推导经济订货批量和补货点的理论计算公式，将抽象的管理目标转化为可量化的数学表达。

在系统设计阶段，以数学模型为指导，将目标函数中的每一个变量映射到具体的数据表字段中——库存量对应inventory表的quantity字段，安全库存对应material表的safety_stock字段，补货点对应material表的reorder_point字段，经济订货批量对应material表的eoq字段，增量与出量对应inventory_transaction表的quantity正负记录。这种"模型先行、数据表跟随"的设计方法，保证了系统从设计之初就内嵌了管理优化的逻辑。

在系统实现阶段，采用前后端分离的开发模式：后端基于Spring Boot 3.2.5构建RESTful API，利用MyBatis-Plus实现数据持久化与乐观锁并发控制，通过JWT和自定义角色注解实现接口级权限管控，借助AOP切面实现操作日志的自动记录；前端基于Vue 3和Element Plus构建用户界面，利用ECharts完成仪表板趋势图和分类统计饼图的可视化展示。

最后，通过功能测试、业务流程验证和数据一致性检查，评估系统在单据流转、库存同步、权限控制和报表统计等方面的表现，验证系统设计的可行性和实用价值。

### 1.5 研究方法

本文综合运用了以下五种研究方法：

文献研究法是本文的基础工作方法。通过查阅仓储管理、库存控制理论以及信息系统建设相关的国内外文献，系统梳理了WMS领域的研究现状、经典模型与技术发展趋势，总结出现有研究的成果与不足，从而明确本文的研究定位——面向中小企业的、模型驱动的轻量化仓储管理系统。

需求分析法贯穿于系统设计的初期阶段。通过调研A公司物资管理的实际业务流程，深入理解不同角色（系统管理员、仓库管理员、采购员、销售员）的工作职责与使用场景，识别出库存数据一致性、业务流程标准化、管理决策可视化三类核心痛点，并据此确定系统的功能边界和设计目标。

数学建模法是本文区别于一般信息系统开发的核心特色。在需求分析的基础上，本文建立了以总库存成本最小化为目标函数的优化模型，涵盖持有成本、订货成本和缺货成本三个维度，并推导了经济订货批量、安全库存和补货点的理论计算公式。该模型不仅是论文的理论组成部分，更是后续数据库表结构设计和业务逻辑实现的核心依据。

软件实现法是将设计转化为可运行系统的工程手段。本系统采用前后端分离的B/S架构，后端基于Spring Boot框架和MyBatis-Plus持久层框架实现RESTful API，前端基于Vue 3框架和Element Plus组件库构建用户界面，利用ECharts完成数据可视化，通过JWT和RBAC模型实现身份认证与权限控制，借助AOP切面实现操作行为的自动审计。

测试验证法用于评估系统的功能完整性和业务正确性。本文采用黑盒测试为主的方法，围绕登录鉴权、基础数据维护、入库业务流程、出库库存控制、库存预警准确性、盘点差异调整和报表统计展示等核心功能设计了测试用例，通过页面表现、接口返回数据与数据库记录的交叉核对，验证系统在功能、流程和数据一致性方面是否达到预期目标。

### 13.2 第一章修改：研究背景与意义重构（三级标题）

#### 1.1 研究背景

##### 1.1.1 仓储管理在企业运营中的重要地位

在制造、商贸、工程及运维等类型企业中，物资仓储管理是连接采购供应、生产制造和销售交付的核心枢纽，其运行质量直接影响企业的资金占用水平、供应链响应速度和整体运营成本。仓储并非单纯的物资存放环节，而是企业资源流转过程中的关键控制节点——每一次入库代表着采购支出的兑现和库存资金的增加，每一次出库关系到生产保障或客户需求的满足，每一次库存决策都在持有成本与缺货风险之间做出权衡。

从资金占用的角度看，库存物资占用了企业大量的流动资金。制造业企业的原材料、在制品和产成品库存合计通常占流动资产的30%至50%。库存水平过高，意味着大量资金沉淀在仓库中无法周转；库存水平过低，又可能导致生产中断或交货延迟。仓储管理的核心挑战，就在于找到资金占用与服务保障之间的最优平衡点。

从信息流的角度看，仓储是采购、生产和销售三大业务环节的信息交汇中心。采购部门需要掌握库存余量来制定补货计划，生产部门需要确认物料齐套性来安排排产，销售部门需要核查库存可用量来向客户承诺交付周期。如果仓储环节提供的数据不及时、不准确，整条供应链的协同效率都将受到严重制约。

从管理成本角度看，仓储运营涉及人力成本（仓库人员的日常作业）、空间成本（库房租赁或折旧）、损耗成本（物资过期、损坏、丢失）以及管理成本（盘点、对账、统计）。缺乏系统化管理的仓库，往往是企业运营成本看不见的"黑洞"——仓储效率每降低一个水平，都会通过物料短缺、生产延误和客户投诉等方式转化为实实在在的财务损失。

正因如此，仓储管理信息化一直是企业数字化转型的重点领域之一。将仓储业务从手工台账提升为信息系统管理，不仅意味着记录方式的改变，更是管理理念的升级——从被动记录库存数量，转向主动控制库存流动、优化库存结构、防范库存风险。

##### 1.1.2 中小企业仓储管理的普遍困境

尽管仓储管理在企业运营中占据如此关键的位置，但广大中小企业在这方面的实际情况却不容乐观。与资金充裕、IT团队完善的大型企业不同，中小企业的仓储管理长期停留在非常初级的水平，面临着与自身业务规模不相匹配的管理短板。

以A公司为代表的中小制造企业，物资管理普遍沿用纸质单据登记与Excel电子表格记录相结合的传统方式。仓库管理员每天面对大量的入库、出库、调拨任务，所有操作依赖人工填写和口头沟通，缺乏系统化的操作规范和自动化的数据校验。这种管理方式的脆弱性在日常作业中表现为频繁的低级错误——入库数量抄写遗漏导致账面少记，出库记录延迟更新导致库存虚高，同一种物资的不同批次记录混淆导致追溯困难。供应商的送货单、仓管员的入库单和财务部门的结算数据来源于不同载体，三方对账时数据不一致的情况经常发生，却很难定位到哪个环节出了差错。

库存账实不符是中小企业仓储管理中最棘手的问题之一。受限于人力物力，很多中小企业难以做到高频次的循环盘点，通常只在季度末或年末进行一次全面盘点。由于日常出入库记录的不断累积偏差，盘点时账实差异往往已经相当可观。面对差异数据，管理者几乎无从追溯——究竟是入库少记了还是出库多记了，是保管过程中发生了损耗还是记录本身出错，一概无法查证。账实不符直接导致采购计划失真、生产排程落空和客户订单交付延迟，给企业带来连锁性的经营风险。

管理效率的低下同样制约着中小企业的发展。采购、仓库、销售三个部门各自维护独立的台账，信息共享全靠电话询问或当面沟通。采购员想要了解某种物资的当前库存，必须等仓库管理员翻找记录后才能得到答复；销售经理想要评估某个订单能否按时交货，同样需要经过多层人工中转。在业务高峰期，这种以人传人的信息流转方式成为制约整体运营效率的瓶颈。更为严重的是，由于出入库数据分散在历年的纸质单据和Excel文件中，管理者无法对库存数据进行任何有价值的统计分析——哪些物资长期积压、哪些物资即将缺货、库存周转是否健康、资金占用是否合理，这些关键问题始终缺乏数据化的答案。管理决策长期依赖个人经验而非系统分析，在面对市场变化时缺乏快速响应的数据能力。

综上所述，中小企业在仓储管理领域面临的困境并非单一的技术问题，而是一个由"人工操作易出错、库存数据不准确、管理决策缺依据"三个层面交织而成的系统性问题。要有效解决这些问题，不能仅靠增加人手或加强员工培训来修补，而需要从根本上变革仓储管理的方式——利用信息技术将人、物、仓、单据和流程纳入统一的管理系统，通过标准化的业务流程、自动化的数据同步和智能化的分析预警，实现仓储管理从"经验驱动"向"数据驱动"的转变。

#### 1.2 研究意义

##### 1.2.1 提升仓储管理效率与决策水平

中小企业的仓储管理长期受困于手工操作带来的效率瓶颈、数据差错和决策失据。入库登记、出库开单、库存查询、报表编制均依赖人工逐条录写，不仅耗时费力，而且难以避免漏记、错记和重复记账等人为失误。出入库记录虽然被保存下来，但以分散、静态的形式堆积在纸质单据和Excel文件中，管理者无法将其转化为有分析价值的经营信息。哪些物资长期积压、哪些物资即将缺货、库存周转是否健康、资金占用是否合理——这些关键问题始终缺乏数据化的答案。

引入仓储管理系统后，基础数据在系统中统一维护，业务单据通过结构化表单录入，系统自动完成数据校验、状态流转和库存更新，每一笔库存变动以流水形式在数据库事务中同步记录，形成了"单据驱动库存变化、流水追溯变动来源"的完整闭环。物资编码唯一性校验、出库审核库存充足性检查、乐观锁并发控制等机制从程序设计层面预防了人为差错，在根源上改善了长期困扰中小企业的账实不符问题。

在管理决策层面，系统以数据库为统一的数据载体，将分散的业务记录整合为可分析的结构化数据资源。安全库存预警将缺货风险的识别从事后补救前移到事前预防；ABC分类分析帮助管理者按库存价值锁定高价值的关键物资；库存周转率从流动效率角度量化资金使用效果；补货点机制基于日均需求量和补货提前期计算补货触发水位，使补货决策从经验判断走向模型驱动。这些功能共同将仓储管理系统从简单的事务记录工具提升为具有经营分析能力的决策辅助平台。

##### 1.2.2 为中小企业信息化建设提供参考

从技术实现的角度看，本文所构建的仓储管理系统采用当前业界广泛使用且高度成熟的开源技术栈——Spring Boot后端框架、Vue 3前端框架、MyBatis-Plus持久层、JWT认证和MySQL数据库。这些技术组合在有限的开发周期和较低的成本约束下完成了从功能设计到上线部署的全过程，充分证明了中小企业无需依赖昂贵的商业软件也能构建符合自身需求的仓储管理系统。

从方法论的角度看，本文区别于一般信息系统开发的核心特色在于"模型驱动设计"的思路。系统不是简单地用软件界面替代纸质表单，而是在设计之初就建立了以总库存成本最小化为目标函数的优化模型，将经济订货批量、安全库存、补货点等库存控制理论中的核心变量直接映射为数据库表的字段。数据库不只是存储数据的容器，而是数学模型的工程化表达。这种"管理理论→数学模型→数据表设计→功能实现"的路径，为中小企业开展其他管理领域的信息化建设提供了可借鉴的方法参考。

从应用前景看，本系统以A公司的实际业务场景为需求来源，覆盖了入库、出库、调拨、报损、盘点和统计分析等完整的管理闭环，RBAC角色权限模型反映了中小企业典型的岗位分工（管理员、仓库管理员、采购员、销售员），具备在同类企业中直接复用或灵活定制的条件，具有较好的推广应用价值。

### 13.5 第二章完整版：仓储管理核心理论（数学方法）


#### 2.1 仓储管理核心理论

##### 2.1.1 库存流量模型：存量-增量-余量

仓储管理的本质是对物资流动状态的动态记录与控制。从生产运作管理的角度，库存变化遵循以下基本流量等式：

$$S_t = S_{t-1} + \Delta_{in} - \Delta_{out}$$

式中，S_t 为 t 时刻的库存存量，Δ_in 为统计期内入库增量，Δ_out 为统计期内出库减量。三个变量分别对应数据库中的核心字段：

| 概念 | 符号 | 含义 | 系统数据库对应 |
|------|------|------|---------------|
| 存量 | S_t | t时刻的实际库存数量 | inventory.quantity |
| 增量 | Δ_in | 入库、调拨入、盘盈之和 | inventory_transaction (change_type='IN') |
| 出量 | Δ_out | 出库、调拨出、报损、盘亏之和 | inventory_transaction (change_type='OUT') |
| 锁定量 | L | 已审核未完成出库的预占数量 | inventory.lock_quantity |
| 可用库存 | S_avail | 实际可分配的库存余量 | quantity − lock_quantity |

增量和出量的来源细化如下：

$$\Delta_{in} = \text{入库完成量} + \text{调拨入量} + \text{盘盈量}$$

$$\Delta_{out} = \text{出库完成量} + \text{调拨出量} + \text{报损报废量} + \text{盘亏量}$$

在实际业务中，可用库存是出库审核的前置判断条件：当且仅当 S_avail ≥ 需求量时，出库单才允许通过审核。系统通过 inventory_transaction 表记录每一次库存变动，使存量S_t在任意时刻都等于初始库存与所有变动记录之和，从而构成了数据库层面库存一致性校验的核心依据。

##### 2.1.2 总成本目标函数与经济订货批量

仓储管理的核心矛盾在于库存水平的选择——库存过高则资金被大量占用，库存过低则面临缺货风险。为将这一权衡转化为可量化的决策依据，本文构建以年度总库存成本最小化为目标函数的优化模型：

$$\min TC = \sum_{i=1}^{n} \left( C_{h,i} \cdot \frac{Q_i}{2} + C_{o,i} \cdot \frac{D_i}{Q_i} + C_{s,i} \cdot E[shortage_i] \right)$$

式中各变量含义如下：

| 变量 | 含义 | 单位 | 系统数据来源 |
|------|------|------|-------------|
| TC | 年度总库存成本 | 元/年 | 系统计算，作为优化目标 |
| Q_i | 物资i的单次订货批量 | 件 | 决策变量，由EOQ公式求解 |
| D_i | 物资i的年需求量 | 件/年 | 统计已完成出库汇总 |
| C_h,i | 单位物资的年持有成本 | 元/(件·年) | material.holding_cost（用户设定） |
| C_o,i | 单次订货成本 | 元/次 | material.order_cost（用户设定） |
| C_s,i | 单位缺货惩罚成本 | 元/件 | 系统配置参数 |
| E[shortage_i] | 期望缺货量 | 件 | 统计值 |

目标函数包含三个成本分量：持有成本 C_h·Q/2 表示平均库存占用的资金成本，与订货批量Q成正比；订货成本 C_o·D/Q 表示全年订货次数的总费用，与Q成反比；缺货成本 C_s·E[shortage] 表示库存不足导致的损失。三者之间存在此消彼长的权衡关系。

对持有成本和订货成本之和关于Q求一阶导数并令其为零：

$$\frac{d}{dQ}\left( C_h \cdot \frac{Q}{2} + C_o \cdot \frac{D}{Q} \right) = \frac{C_h}{2} - \frac{C_o D}{Q^2} = 0$$

解得经济订货批量公式：

$$EOQ = Q^* = \sqrt{\frac{2 D C_o}{C_h}}$$

EOQ值存储在 material.eoq 字段中，作为系统推荐订货批量的理论基础。当用户未设置成本参数时，系统默认EOQ取安全库存的两倍。

##### 2.1.3 安全库存与订货点模型

经济订货批量确定了每次订货的最优数量，但并未回答"何时订货"的问题。在实际运营中，物资需求存在随机波动，供应商交货也存在不确定性。为确保在需求波动和供应延迟的情况下仍能满足生产或销售需要，企业需要设置安全库存作为缓冲。

安全库存的计算公式为：

$$SS = Z \times \sigma_d \times \sqrt{L}$$

式中，Z 为服务水平系数（95%服务水平对应 Z = 1.65，表示允许5%的缺货概率），σ_d 为日需求量的标准差（通过对历史出库数据的统计得到），L 为平均补货提前期（从发出订单到物资入库所需的平均天数）。计算结果存储在 material.safety_stock 字段中。

订货点确定触发补货的最低库存水位。当库存量降至订货点以下时，系统应建议发起一次新的采购补货。订货点的计算公式为：

$$ROP = \bar{d} \times L + SS$$

式中，d̄ 为日均需求量（系统通过统计最近90天的已完成出库量除以90得到，存储在material.daily_demand字段中），L 为补货提前期（存储在 material.lead_time 字段，默认7天），SS 为安全库存。ROP值存储在 material.reorder_point 字段中。

安全库存与订货点的关系需要区分清楚：安全库存SS是应对不确定性的"最后防线"，库存一般不应低于此线；订货点ROP是触发补货行动的"预警线"，该值高于安全库存，高度由补货提前期内的预期需求（d̄×L）决定。当库存降至ROP时发出补货订单，在补货到货前的L天内，库存继续消耗，安全库存负责在这段时间内吸收需求波动。

##### 2.1.4 ABC分类法与库存周转率

ABC分类法基于帕累托法则（即20%的物资占有80%的库存价值），将物资按库存资金占用程度分为A、B、C三类，以便企业实施差异化管理策略。具体计算步骤如下：

第一步，计算每种物资的库存价值：V_i = quantity_i × unit_price_i。

第二步，按V_i从大到小降序排列。

第三步，计算每种物资的累计价值占比：P_k = Σ_{i=1}^k V_i / Σ_{i=1}^n V_i × 100%。

第四步，按P_k划分等级：累计占比不超过80%的物资归为A类（高价值物资，需重点管控）；80%至95%之间的归为B类（中等价值，常规管理）；超过95%的归为C类（低价值，简化管理）。

库存周转率衡量物资的流动效率，计算公式为：

$$R = \frac{Q_{out}}{\bar{I}}$$

式中，Q_out 为统计期内已完成的出库总量，Ī 为同期平均库存量，通常取期初库存与期末库存的算术平均。周转率R值越高，表明物资流动越快，库存资金占用效率越好；反之则可能存在物资积压问题。系统支持按日期范围灵活查询各物资的周转率，帮助管理者识别滞销或积压物资，为库存结构优化提供定量依据。

以上五组数学模型共同构成了本文仓储管理系统的理论核心。它们之间的关系是：库存流量公式定义了存量的变化规律，目标函数给出了库存优化的方向和评价标准，EOQ回答了"订多少"的问题，安全库存和订货点回答了"何时订"的问题，ABC分类法和库存周转率则为物资分级管理和效率评估提供了分析工具。此外，上述模型中涉及的每一个变量——S_t、Δ_in、Δ_out、SS、ROP、EOQ、D、C_h、C_o、L——均在后文数据库设计中对应到具体的数据表字段，实现了管理理论与软件工程的统一。

**基础流量等式**：

其中各项定义如下：

| 概念 | 符号 | 含义 | 数据库对应 |

| 存量 | S_t | t时刻的库存持有量 | inventory.quantity |

**流量来源细化**：

$$\Delta_{in} = \text{入库完成量} + \text{调拨入量} + \text{盘盈量}$$

$$S_{avail} = S_t - L \geq \text{需求量}$$

**存量守恒验证**（数据库一致性检查的核心依据）：
对于任意仓库中的任意物资，其当前库存量必须满足：
$$inventory.quantity = \text{初始库存} + \sum inventory\_transaction.quantity$$

该流量模型将抽象的库存变化过程映射到具体的数据表字段，使业务流程设计有了明确的数学依据。例如，出库审核阶段的库存检查（checkStockSufficient）直接对应 S_avail ≥ 需求量 的判定；入库完成和出库完成分别对应 Δin 和 Δout 的写入操作。

#### 3.1 系统优化目标与数学模型

仓储管理的核心矛盾在于：库存过高会占用资金，库存过低会导致缺货。因此，系统设计的根本目标是在**满足服务水平约束的前提下，最小化总库存成本**。

##### 3.1.1 总成本目标函数

$$\min TC = \sum_{i=1}^{n} \left( C_{h,i} \cdot \frac{Q_i}{2} + C_{o,i} \cdot \frac{D_i}{Q_i} + C_{s,i} \cdot E[shortage_i] \right)$$

**变量定义**：

| 变量 | 含义 | 单位 | 在本系统中的数据来源 |
|------|------|------|---------------------|
| TC | 总库存成本 | 元/年 | 优化目标，由系统计算 |
| Q_i | 物资i的订货批量 | 件 | 决策变量，由EOQ公式计算 |
| D_i | 物资i的年需求量 | 件/年 | Σ出库明细(一年的已完成出库) |
| C_h,i | 物资i的单位年持有成本 | 元/(件·年) | material.holding_cost |
| C_o,i | 物资i的单次订货成本 | 元/次 | material.order_cost |
| C_s,i | 物资i的单位缺货惩罚成本 | 元/件 | 可配置参数（论文中取物资单价的2倍）|
| E[shortage_i] | 物资i的期望缺货量 | 件 | 统计值 = max(0, 需求量 - 可用库存) |

##### 3.1.2 约束条件

$$Q_i \geq 0, \quad S_{t,i} \geq 0, \quad S_{avail,i} \geq 0$$

$$S_{t,i} \geq SS_i \quad \text{(安全库存约束，即存量不低于安全库存)}$$

$$S_{t,i} + Q_i \leq \text{max\_stock}_i \quad \text{(最大库存约束)}$$

##### 3.1.3 EOQ（经济订货批量）推导

对目标函数中持有成本与订货成本之和求导，令导数为零：

$$\frac{d}{dQ}\left( C_h \cdot \frac{Q}{2} + C_o \cdot \frac{D}{Q} \right) = \frac{C_h}{2} - \frac{C_o D}{Q^2} = 0$$

解得：

$$EOQ = Q^* = \sqrt{\frac{2 D C_o}{C_h}}$$

该值即为使单物资年库存总成本最小的订货批量，存储在 material.eoq 字段中。

##### 3.1.4 安全库存（SS）计算

$$SS = Z \times \sigma_d \times \sqrt{L}$$

其中 Z 为服务水平系数（95%服务水平取1.65），σ_d 为日需求标准差（由历史出库数据统计），L 为平均补货提前期。计算结果存储在 material.safety_stock 字段中。

##### 3.1.5 补货点（ROP）计算

$$ROP = \bar{d} \times L + SS$$

其中 d̄ 为日均需求量（最近90天出库总量/90），L 为补货提前期，SS 为安全库存。

**补货触发条件**：当前库存量 ≤ ROP 时，系统生成补货建议。

**补充说明**：补货点不同于安全库存。安全库存是保障库存的"底线"，而补货点是触发补货行为的"预警线"。当库存降至ROP时，发出补货订单；在补货到货前的L天内，库存继续消耗，安全库存SS负责吸收这段时间内的需求波动。

### 13.8 第四章新增：补货点机制（Reorder Mechanism）

#### 4.X 补货点机制设计与实现

##### 4.X.1 补货点机制的业务逻辑

补货点机制是将库存控制数学模型直接嵌入系统的核心功能。其业务逻辑如下：

1. **需求统计**：系统从 inventory_transaction 表中统计最近90天每种物资的已完成出库总量，除以90得到日均需求量 d̄。
2. **ROP计算**：ROP = d̄ × 提前期L + 安全库存SS。提前期默认取7天，可在物资管理中配置；安全库存已存储在material.safety_stock中。
3. **EOQ计算**：EOQ = √(2DS/H)，其中D = d̄ × 365（年需求量），S = material.order_cost，H = material.holding_cost。若未设置成本参数，默认EOQ = 安全库存 × 2。
4. **补货判断**：遍历所有库存记录，若当前库存 ≤ ROP，则生成补货建议并展示紧急程度。
5. **建议输出**：表格展示物资名称、当前库存、安全库存、补货点、EOQ建议补货量、缺口数量及紧急程度标签。

##### 4.X.2 紧急程度分级

| 条件 | 等级 | 标签颜色 |
|------|------|---------|
| currentStock ≤ 0 | 严重缺货 | 红色 (danger) |
| 0 < currentStock ≤ safetyStock | 紧急补货 | 橙色 (warning) |
| safetyStock < currentStock ≤ ROP | 建议补货 | 蓝色 (primary) |

```java
            Map<String, Object> item = new LinkedHashMap<>();
            item.put("materialName", material.getName());
            item.put("currentStock", currentQty);
            item.put("safetyStock", safetyStock);
            item.put("reorderPoint", reorderPoint);
            item.put("eoq", eoq);
            // 建议补货量 = max(EOQ, ROP - 当前库存 + 安全库存)
            item.put("suggestedQuantity", 
                Math.max(eoq, reorderPoint - currentQty + safetyStock));
            item.put("shortage", Math.max(0, reorderPoint - currentQty));
            // 紧急程度判断
            String urgency = currentQty <= 0 ? "严重缺货" :
                currentQty <= safetyStock ? "紧急补货" : "建议补货";
            item.put("urgency", urgency);
            result.add(item);
        }
    }
    return result;
}

public void recalculateReorderParams() {
    for (Material m : materialMapper.selectList(null)) {
        double dailyDemand = calculateDailyDemand(m.getId());  // 90天平均
        int leadTime = m.getLeadTime() != null ? m.getLeadTime() : 7;
        int safetyStock = m.getSafetyStock() != null ? m.getSafetyStock() : 0;
        
        // ROP = d̄ × L + SS
        int rop = (int) Math.ceil(dailyDemand * leadTime) + safetyStock;
        
        // EOQ = sqrt(2DS/H)
        int eoq = 0;
        if (m.getOrderCost() != null && m.getHoldingCost() != null 
            && m.getHoldingCost().compareTo(BigDecimal.ZERO) > 0 && dailyDemand > 0) {
            double annualD = dailyDemand * 365;
            eoq = (int) Math.round(
                Math.sqrt(2 * annualD * m.getOrderCost().doubleValue() 
                          / m.getHoldingCost().doubleValue()));
        }
        
        // 更新material表
        Material update = new Material();
        update.setId(m.getId());
        update.setDailyDemand(BigDecimal.valueOf(dailyDemand));
        update.setReorderPoint(rop);
        update.setEoq(eoq);
        materialMapper.updateById(update);
    }
}
```

## 十五、第二章 仓储管理的数学方法（修改版——新增流量模型）

以下是修改后的完整内容，将原来的(1)-(5)改为(1)-(6)，在EOQ之前新增了库存流量模型：

---

1. 仓储管理的数学方法

以下数学模型为本文系统的库存预警、ABC分析及周转率统计提供了定量依据。

（1）库存流量模型：存量-增量-余量

仓储管理本质上是对物资流动状态的动态记录与控制，库存变化遵循基本流量等式：

$$S_t = S_{t-1} + \Delta_{in} - \Delta_{out}$$

其中，S_t为t时刻的库存存量，Δ_in为统计期内入库增量（含入库完成、调拨入和盘盈），Δ_out为统计期内出库减量（含出库完成、调拨出、报损报废和盘亏）。在此基础上定义可用库存余量S_avail = S_t − L，其中L为已审核未完成出库的锁定库存量，该余量作为出库审核判断的前置条件：仅当S_avail ≥ 需求量时，系统才允许通过审核。该模型将库存变化过程映射到具体的数据表字段——inventory表存储存量S_t和锁定量L，inventory_transaction表记录每一次Δ_in和Δ_out的变动细节，并保证任意时刻inventory.quantity = 初始库存 + Σinventory_transaction.quantity。

（2）经济订货批量模型（EOQ）

经济订货批量确定使年库存总成本最小的订货量，其公式为：

$$EOQ = \sqrt{\frac{2DS}{H}}$$

其中，D为年需求量，S为单次订货成本，H为单位年持有成本。该模型用于指导合理补货批量。

（3）安全库存模型

为应对需求波动和供应不确定性，安全库存计算公式为：

$$SS = Z \times \sigma_d \times \sqrt{L}$$

其中，Z为服务水平系数，σ_d为日需求标准差，L为平均补货提前期。当需求与提前期均不确定时，采用双变量公式：

$$SS = Z \times \sqrt{\sigma_d^2 \times L + \sigma_L^2 \times \bar{d}^2}$$

本文系统将物资表中的safety_stock字段设置为SS的计算值，并实时判断当前库存Q是否满足Q < SS以触发预警。

（4）ABC分类法

依据帕累托法则，将物资按库存价值划分为A、B、C三类。计算步骤为：首先计算每种物资的年消耗价值V_i = 需求量i × 单价i；其次按V_i降序排序，计算累计价值占比P_k = Σ_{i=1}^k V_i / Σ_{i=1}^n V_i；最后按P_k ≤ 80%为A类（高价值）、80% < P_k ≤ 95%为B类（中等价值）、P_k > 95%为C类（低价值）进行划分。

（5）库存周转率模型

库存周转率衡量物资流动效率，公式为：

$$R = \frac{Q_{out}}{\bar{I}}$$

其中，Q_out为期间出库总量，Ī为期间平均库存（通常取期初与期末库存的平均值）。R值越高，表明库存周转越快，资金占用效率越高。

（6）订货点模型

订货点确定触发补货的库存水位，公式为：

$$ROP = \bar{d} \times L + SS$$

其中，d̄为平均日需求量，L为补货提前期，SS为安全库存。当当前库存降至ROP以下时，系统应建议补货。

以上数学模型构成了本文系统库存管理功能的理论核心。

## 十六、第三章新增：系统优化目标与数学模型（加在需求分析之前）


### 3.1 系统优化目标与数学模型

在展开需求分析之前，首先需明确系统的设计目标。仓储管理的核心矛盾在于库存水平的两难取舍：库存量过高，大量流动资金被占用；库存量过低，则面临缺货停工和交付延迟的风险。因此，系统设计不能仅停留在记录物资出入库的事务处理层面，而应追求一个可量化的优化目标——在满足生产所需服务水平的前提下，使总库存成本最小化。

#### 3.1.1 总成本目标函数

将上述目标转化为数学表达，对物资i（i = 1, 2, …, n），设其单次订货批量为Q_i，建立年度总库存成本最小化模型：

$$
\min\ TC = \sum_{i=1}^{n} \left[ 
C_{h,i} \cdot \frac{Q_i}{2} 
+ C_{o,i} \cdot \frac{D_i}{Q_i} 
+ C_{s,i} \cdot E(shortage_i)
\right]
$$

目标函数由三项成本构成：

**持有成本** C_h·Q/2。每次订货量为Q，在均匀消耗假设下，平均库存量为Q/2，乘以单位物资的年持有成本C_h（含资金占用利息、仓储空间费用、保险费及变质损耗等）。该项与订货批量Q成正比——每次进货越多，平均库存越高，持有成本越大。

**订货成本** C_o·D/Q。年需求量D除以订货批量Q得到全年订货次数，乘以单次订货成本C_o（含运输费、装卸费、检验费及订单处理费等）。该项与订货批量Q成反比——每次进货越多，全年订货次数越少，订货总成本越低。

**缺货成本** C_s·E(shortage)。当库存不能满足需求时，企业承受停工待料、紧急高价采购或客户流失等损失。C_s为单位缺货惩罚成本，E(shortage)为期望缺货量。

三项成本此消彼长：增大订货批量可以降低订货成本和缺货风险，但会抬高持有成本；减小订货批量可以压缩持有成本，但会增加订货频率和缺货概率。目标函数的核心在于找到使三者和最小的最优订货批量。

#### 3.1.2 经济订货批量的推导

缺货成本在实际运营中较难精确量化，通常转化为服务水平约束单独处理。在此条件下，目标函数简化为持有成本与订货成本的权衡。对两项之和关于Q求一阶导数并令其为零：

$$
\frac{d}{dQ} \left[ C_h \cdot \frac{Q}{2} + C_o \cdot \frac{D}{Q} \right]
= \frac{C_h}{2} - \frac{C_o D}{Q^{2}} = 0
$$

移项整理：

$$
\frac{C_h}{2} = \frac{C_o D}{Q^{2}}
\quad\Rightarrow\quad
Q^{2} = \frac{2 D C_o}{C_h}
$$

解得经济订货批量（Economic Order Quantity, EOQ）：

$$
Q^{*} = \sqrt{\frac{2 D C_o}{C_h}}
$$

EOQ是使年度持有成本与订货成本之和最小的最优订货量。在本系统中，年需求量D通过对已完成出库记录的年度汇总统计得到，单位持有成本C_h和单次订货成本C_o由用户在物资管理界面中设置，分别存入material表的holding_cost字段和order_cost字段，系统据此自动计算EOQ并存入material表的eoq字段。当用户尚未配置成本参数时，系统默认将EOQ取为安全库存的两倍，以保证基本的补货参考值。

#### 3.1.3 约束条件与模型落地的数据映射

系统的库存运作受以下约束：库存量非负（S_t ≥ 0）；库存量不低于安全库存以维持服务水平（S_t ≥ SS）；库存量加本次补货量不超过仓库最大容量（S_t + Q ≤ max_stock）。这些约束在系统中分别对应出库扣减时的库存充足性检查、库存预警的安全库存判定和入库完成时的仓库容量校验。

上述优化模型不仅提供理论框架，更直接决定了本系统的数据库结构。模型中的每一个变量均被映射为具体的数据表字段：

存量S_t对应inventory.quantity，由inventory_transaction表中历次出入库流水的代数和进行一致性验证。订货批量Q对应material.eoq，由EOQ公式计算得到。年需求量D通过对outbound_item表已完成的出库记录按年度汇总统计。持有成本C_h和订货成本C_o分别对应material.holding_cost和material.order_cost。安全库存SS对应material.safety_stock，由用户或系统根据Z·σ_d·√L计算设置。日均需求量d̄对应material.daily_demand，提前期L对应material.lead_time（默认7天），二者与SS共同决定补货点ROP = d̄·L + SS，存储在material.reorder_point字段中。

通过以上映射，数学模型与数据库设计形成一一对应的关系——优化目标不再停留于理论层面，而是嵌入到系统每次计算补货建议的代码逻辑中，使系统从被动记录走向主动辅助决策。
